Aletheia: The AI Assistant That Can Learn, Unlearn, and Prove It

Built with self-hosted open-source Cognee, Aletheia turns memory into something visible, auditable, and correctable.

Cognee Hackathon · Best Blogs Side Track · July 2026

01 The problem nobody talks about

Every RAG assistant you have ever used can learn. You give it a document, it chunks it, embeds it, and retrieves it later. That part works. People have been solving retrieval for years.

The hard part is what happens when the document is wrong.

A study gets retracted. A policy updates. An internal memo supersedes last month's guidance. In most AI assistants, the outdated information is still in the vector store, still retrievable, still confidently cited in answers. The system learned, but it never unlearned. It does not even know unlearning is something it should do.

The hard part was not making the AI remember. The hard part was making it change its mind.

Aletheia is a research assistant built around that specific problem. Not a general-purpose chatbot. Not a document Q&A demo. A system that treats memory as a lifecycle — something that can be created, trusted, questioned, revoked, and audited — and makes every stage of that lifecycle visible to the person relying on it.

02 The idea: memory as a lifecycle

Most AI memory is a blob. You put things in. They stay in. If you are lucky, there is a delete button somewhere. But deletion is not unlearning — it is just removing text. The system does not know why it forgot, what changed, or what it should answer differently now.

Aletheia's design starts from four operations that every piece of knowledge should go through:

📥
RememberIngest a source
🔍
RecallAnswer from trusted sources
⚠️
ConflictDetect contradiction
🗑
ForgetRetract discredited data
🔄
ImproveRe-enrich survivors
✦
AnswerDifferent answer, different citations

These are not abstract concepts. Each one maps directly to a Cognee operation. That is not an accident — it is the reason I chose Cognee in the first place.

03 How Cognee became the product

Cognee is often described as a memory framework. That is technically accurate and completely understates what happened when I started building Aletheia. Cognee is not a backend service that Aletheia wraps with a UI. Cognee's operations are the product's operations. Every feature the user sees maps to a Cognee call.

The entire system runs on self-hosted, open-source cognee==1.2.2. No Cognee cloud account. No hosted API key. All data — SQLite, LanceDB vectors, the graph store — lives in a local ./data/ directory.

remember()

Every source is ingested into its own Cognee dataset — src_<slug>. This is the foundation: one dataset per source means each source can be trusted or revoked independently.

recall()

When Aletheia answers a question, it passes only trusted dataset names to recall(). Retracted sources are not hidden — they are structurally outside the recall scope.

forget()

The retraction itself. forget(dataset=...) surgically deletes one source's memories from the graph. This is a real deletion, not a soft flag.

improve()

After a retraction, the remaining trusted datasets are re-enriched. The graph re-settles into a corrected worldview instead of just having a hole where the retracted data was.

get_graph_data()

The constellation UI — every node, every link — is pulled directly from Cognee's graph. No mockup layer. The graph is the visualization.

In Aletheia, forgetting is not a UI trick. It is a Cognee dataset operation.

04 The "un-forgetting" moment

Here is the demo narrative. All sources are fictional — no real brands, journals, or people are referenced.

Three sources are loaded: a fictional supplement-industry trial claiming a 40% memory boost from a compound called "Mnemosyne-7," a news piece amplifying the claim, and a university meta-analysis that is more cautious.

Ask: "Does Mnemosyne-7 improve memory? What actually works?"

Aletheia answers, and the answer parrots the 40% claim. It cites the trial and the news piece. So far, this looks like every other RAG assistant.

Then a retraction notice arrives. One click ingests a journal retraction into the system — live, through the same remember() pipeline as every other source.

Aletheia reads the retraction. It detects — on its own — that this new source disputes a source it currently trusts. An amber conflict card slides in, naming the disputed source and explaining the contradiction. The human confirms the retraction.

What happens next is the point of the whole project: the retracted source's nodes ignite red in the constellation and die. A live ticker counts: "10 memories removed · 20 links severed." Those numbers are measured from the graph — a before/after diff of real nodes, not hardcoded text.

Ask the same question again. The answer now says the claim is retracted and unsupported, recommends what the surviving meta-analysis actually found (exercise and sleep), and cites different sources. The AI visibly changed its mind.

05 Why dataset-scoped trust matters

You could "retract" a source in a vanilla RAG system by deleting its chunks from the vector store. But that is a blunt operation. You lose the context of why it was removed. You cannot audit it. And if the source's claims were repeated or summarized elsewhere in the embeddings, fragments survive.

Design principle

Trust is a recall filter. In Aletheia, trust is not a metadata tag — it determines what recall() can even see. Retracted knowledge is not buried lower in the results. It is removed from the space the assistant is allowed to recall from.

This is possible because of a specific design decision: one Cognee dataset per source. When you call recall(datasets=[...]) and only pass trusted dataset names, the retracted source's memories are not just deprioritized — they are unreachable. The recall boundary is structural, not heuristic.

Key distinction

Forgetting is not hiding. A retracted source is not buried lower in the prompt. It is removed from the space the assistant is allowed to recall from. This is a Cognee forget(dataset=...) operation, not a UI filter.

06 Visual memory: the constellation

Aletheia's main interface is a force-directed graph — what we call the constellation. Every star is a real graph node pulled from get_graph_data(). Nodes are colored by source. Links represent relationships Cognee extracted during ingestion. The layout is not decorative. It is the system's actual memory, rendered.

The graph does not just decorate the product. It is the product's memory made visible.

When you click a citation chip in an answer, the contributing nodes pulse. You can trace exactly which memories the answer stands on. This attribution is real — the node IDs come from Cognee's relational node ledger, cross-checked by a graph-node diff taken before and after each ingestion.

During retraction, the constellation is the main event. The retracted nodes ignite, collapse, and disappear over 1.6 seconds while the rest of the graph re-settles under the force simulation. It is not an animation played over the data. It is the data driving the animation.

07 Auditability: the ledger and the diff

Every belief change is recorded in the Mind-Change Log — a JSONL ledger that the UI renders as a scrolling timeline. When you retract a source, a new entry appears: what was retracted, why, and the exact node and link counts that changed.

Implementation detail

The graph diff is measured. Before and after every forget(), Aletheia snapshots the graph via Cognee's node ledger. The "10 memories removed · 20 links severed" numbers in the UI are a subtraction of those two snapshots — not hardcoded, not estimated.

This matters for trust. If an assistant says "I no longer believe X," you should be able to ask: what exactly changed? How many memories were affected? Which sources remain? The ledger answers all of that.

08 The "Stateless LLM" toggle

Aletheia has a built-in comparison mode. Flip the toggle, and the same question is answered by a vanilla LLM call — no memory, no retrieval, no sources. The two answers appear side by side.

Aletheia (Cognee-backed) Stateless LLM
Cites sources ✓ Named, traceable ✗ None
Can retract ✓ Surgical per-source ✗ No memory to retract
Answers change after correction ✓ Verified by test ✗ Same answer every time
Shows what it stands on ✓ Node attribution ✗ Opaque
Accountable ✓ Ledger + graph diff ✗ No record

The stateless answer feels confident. It often sounds right. But it has no receipts. It cannot tell you why it believes something, and it cannot change its mind when the evidence does. The toggle makes this gap visceral — not by arguing for memory, but by showing the absence of it.

09 Testing the claim

Claims about AI behavior are easy to make and hard to verify. That is why Aletheia includes an automated flip test — a pytest that drives the entire demo narrative programmatically:

  1. Seed the three fictional sources via remember()
  2. Ask the question via recall()
  3. Ingest the retraction notice
  4. Assert the conflict was auto-detected
  5. Retract through the same code path as the UI button
  6. Ask again and assert the answer changed
  7. Assert the discredited claim ("40%") is gone from the new answer

10 What I learned building it

I started Aletheia because I was frustrated. I kept reading about AI memory systems that were really just bigger context windows, and I kept seeing demos where "knowledge management" meant "we embedded your PDFs." That felt like it was missing something fundamental about how knowledge actually works — specifically, that knowledge changes, gets corrected, and sometimes needs to be taken back.

When I found Cognee, the thing that caught my attention was forget(). Not as a cleanup utility — as a first-class operation with the same standing as remember(). That implied a worldview where memory is not a warehouse you put things into. It is a living system with a lifecycle. And once you have that, you can build things that most RAG systems cannot even express: trust scoping, retraction, auditable belief changes.

The hardest part was not the code. The hardest part was resisting the temptation to hide complexity. It would have been easy to make retraction a background process — silently update the data and show a cleaner answer next time. But the whole point is that the user should see the forgetting happen. They should see the nodes die, count the severed links, read the ledger entry. If AI memory is going to be trustworthy, it has to be inspectable.

I also learned that dataset-scoped operations are more powerful than they sound. "One dataset per source" is a simple design rule, but it cascades into everything: trust becomes a recall filter, retraction becomes surgical, attribution becomes traceable, and the graph becomes a genuine map of what the system believes and why.

Building with self-hosted Cognee meant I could inspect everything. When something went wrong — a node survived retraction when it should not have, or a shared entity was incorrectly killed — I could look at the actual SQLite database, the actual graph store, and understand what happened. That is the value of open source for a project like this: not just "free," but legible.

11 Closing

Aletheia is a hackathon project. It is not deployed in production. It does not handle real medical, legal, or scientific data. The demo sources are fictional.

But the mechanism is real. Every remember(), every recall(), every forget() runs against live Cognee. The graph nodes are real. The before/after diffs are measured. The flip test passes green.

The question I keep coming back to is: if an AI assistant learns something wrong, what happens next? In most systems, nothing. The wrong information sits there, confidently retrievable. In Aletheia, the contradiction is detected, the retraction is proposed, the human confirms, the graph changes, the ledger records it, and the next answer is different.

AI memory should not only be powerful. It should be correctable.
